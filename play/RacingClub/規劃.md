# RacingClub 規劃

## 1. 目標

本區塊的公開路徑改為：

- `play/`
- `play/RacingClub/`
- `play/RacingClub/TimeAttack/`

整體採用「本機 builder 產出靜態頁與 JSON，再 push 到 GitHub Pages」的模式。

核心原則：

- 單一資料表輸入
- 自動生成所有分析
- 網站只讀 builder 產物，不直接承擔資料編輯
- `RacingClub` 主頁只做入口與社群資訊
- `TimeAttack` 子站承接資料密集的排行與分析

## 2. 公開頁面結構

### `play/`

Project T / Play 的總入口，用來分流：

- Racing Club
- 山道世界索引
- 活動 / 賽季頁

### `play/RacingClub/`

Racing Club 主頁。

第一版只保留：

- 簡介
- 社群連結
- Time Attack 入口
- 後續世界 / 活動 / 媒體接口占位

### `play/RacingClub/TimeAttack/`

Time Attack 資料站入口，底下再分頁承接各種分析。

第一版先做：

- `index.html`：總覽 / 分流頁
- `tracks.html`：賽道分析頁
- `players.html`：車手分析頁
- `vehicles.html`：車輛分析頁
- `events.html`：活動 / 賽季頁
- `review.html`：審核 / 正式紀錄頁

## 3. 資料流

目標資料流：

`Google Sheet -> local export/builder -> play/RacingClub/TimeAttack/data/*.json + 生成圖片 -> git push -> GitHub Pages`

Builder 的責任：

- 讀取輸入表
- 讀取索引 / 對照表
- 計算所有分析頁資料
- 輸出 JSON
- 輸出需要的 PNG / 記錄圖
- 更新靜態頁面用的最後更新時間 / manifest

前端的責任：

- 讀取 `data/manifest.json`
- 讀取各頁對應 JSON
- 呈現排行榜、摘要、分類與導覽

## 4. 輸入表現況

目前輸入表欄位：

- `日期`
- `賽道`
- `路線`
- `賽道作者`
- `玩家名稱`
- `時間`
- `使用車輛`
- `平台`
- `系統`
- `與賽道紀錄相比`
- `原始時間`
- `時間轉換`
- `時間轉換`

現況問題：

- `賽道作者`、`玩家名稱`、`世界資訊`、時間換算值混在輸入表中
- 有些欄位其實是分析層或索引層資料，不應和輸入混在一起
- `賽道` 實際代表的是「賽道世界」，不是抽象路線名稱
- `使用車輛` 目前沒有拆開「世界內變體」與「統計上同車系」

## 5. 建議的單一輸入表欄位

第一版建議輸入表轉為代碼導向，只保留必要輸入：

- `record_date`
- `track_world_code`
- `route_code`
- `player_id`
- `vehicle_variant_code`
- `platform_code`
- `system_code`
- `lap_time_raw`
- `record_channel`
- `review_status`
- `event_code`
- `season_code`
- `submission_note`

說明：

- `track_world_code`：對應賽道世界。因為同一條賽道可能有不同世界、作者、車輛與風格。
- `route_code`：同一個賽道世界下的路線代碼。
- `player_id`：唯一值，對應多個顯示名稱。
- `vehicle_variant_code`：世界內車輛變體代碼。
- `record_channel`：`approved_record` 或 `normal_time_attack`。
- `review_status`：`pending`、`approved`、`rejected`。

輸入表不需要再直接填：

- 賽道標籤
- 世界名稱
- 世界連結
- 賽道作者
- 玩家顯示名稱
- 玩家團隊
- 車輛統計主類
- 賽季名稱
- 活動名稱
- 時間換算值
- 時間顯示值
- 與賽道紀錄相比

這些應由 lookup 表或 builder 補上。

## 7. 正式紀錄與一般紀錄

需要拆兩層概念：

### `record_channel`

母分類，用來表示該筆紀錄想進哪種版面：

- `approved_record`
- `normal_time_attack`

### `review_status`

審核狀態：

- `pending`
- `approved`
- `rejected`

目前規則：

- 目前所有既有資料先視為 `review_status = pending`
- `approved_record` 板只顯示 `record_channel = approved_record AND review_status = approved`
- `normal_time_attack` 板可顯示一般紀錄
- 個人紀錄與車輛紀錄可以吃 `pending` / `normal` 紀錄，但應明確標記

這樣可以同時滿足：

- 有正式賽道紀錄標準
- 不丟掉一般時間攻克資料
- 個人頁 / 車輛頁仍有足夠資料可看

## 8. TimeAttack 前端頁面職責

### `index.html`

顯示：

- 最後更新時間
- 紀錄總數
- 已審核正式紀錄數
- 一般 time attack 紀錄數
- 賽道 / 車手 / 車輛數量
- 子頁面入口

### `tracks.html`

承接：

- 賽道紀錄（每玩家一筆最快）
- 賽道紀錄（每車輛一筆最快）
- 賽道世界列表
- 賽道車種熱門度
- 賽道紀錄頻率
- 賽道紀錄變化

### `players.html`

承接：

- 個人紀錄（每賽道一筆最快）
- 個人成績變化
- 個人使用車種頻次
- 個人使用車輛頻次
- 個人頁入口

### `vehicles.html`

承接：

- 車輛紀錄（每賽道一筆最快）
- 車輛表現和賽道性質的關係
- 世界內變體與統計車系的切換

### `events.html`

承接：

- 活動紀錄
- 錦標賽
- 排位賽
- 賽季積分

備註：

- 這頁會比較複雜，因為要吃活動格式、賽季與積分規則
- 第一版先保留頁面與資料接口

### `review.html`

承接：

- Approved Record 板
- 待審核統計
- 正式紀錄審核規則說明

## 9. Builder 輸出檔案

第一版固定輸出到：

- `play/RacingClub/TimeAttack/data/manifest.json`
- `play/RacingClub/TimeAttack/data/summary.json`
- `play/RacingClub/TimeAttack/data/tracks.json`
- `play/RacingClub/TimeAttack/data/players.json`
- `play/RacingClub/TimeAttack/data/vehicles.json`
- `play/RacingClub/TimeAttack/data/events.json`
- `play/RacingClub/TimeAttack/data/review.json`

之後可擴充：

- `play/RacingClub/TimeAttack/generated/*.png`
- `play/RacingClub/TimeAttack/generated/*.json`

`manifest.json` 要至少包含：

- schema version
- generated_at
- source sheet revision
- page -> data file mapping

## 10. 第一版落地順序

1. 先完成站點骨架與路由

7. 再逐頁補上真實分析
