# 頁尾分析模組規劃(zh-TW)

日期:2026-06-13。範圍:在 TimeAttack 各頁尾端加入分析模組,重點三張詳情頁
(車輛個體 `vehicle.html?id=`、車手個人 `player.html?id=`、賽道個別 `track.html?id=`)
+ 首頁(overview)一個熱門度頻率圖。資料一律來自既有 709 筆紀錄,**不新增任何採集**。

---

## 0. 設計原則

1. **零新資料**:所有圖表用 builder 既有的 records / lookup 重新聚合;使用者不必多輸入。
2. **聚合在 builder、繪圖在前端**:Python 端把每張圖預算成小陣列塞進該頁 JSON;
   前端只負責畫,維持「JSON 餵養靜態頁」架構,也讓圖表邏輯集中在一處好維護。
3. **不引入圖表庫**:用手刻的 inline SVG + CSS 長條(資料量小、每個實體幾十筆),
   風格吃既有 `--ta-*` 暗色系。Leaflet 是地圖的必要例外,圖表不需要再 vendor 一個 Chart.js。
4. **小樣本防呆**:有效紀錄 < 門檻(建議 3)的實體,模組顯示「資料太少」而非畫出誤導的圖。
5. **沿用 `renderModule(zh,en,zhTitle,enTitle,bodyHtml)`** 包裝,接在各 detail 渲染器 return 的尾端。

---

## 1. 既有資料盤點(規劃依據,實作前對照)

每筆 record 可用欄位:`record_date`(ISO)、`lap_time_ms`、badge(TR/CR/PR,
由 `badge_code_for_record`)、`platform_code`、`system_code`、`track_env`、`track_tags`、
`difficulty`、`country/region/locality`、`player_display_name`、`vehicle_model_name`、
`vehicle_variant_name`、`route_key`。

builder 各 detail 函式「已經算出但未全部輸出」的聚合:

| 函式 | 已有 |
| --- | --- |
| `build_player_pages` | `history`(**每條路線的時間序列**:date/lap_time_ms/vehicle/is_pb)、vehicle 使用 Counter、env Counter、tag Counter、badge_counts |
| `build_vehicle_pages` | variant/driver/env/tag Counter、badge_counts、各路線最佳 record_rows |
| `build_track_pages` | 每條路線的 route_rows(該路線全部跑次)、player_rows、vehicle_rows、record_count |
| `build_overview_extras` | track_run_counter、vehicle_run_counter(**完整 Counter,目前只取 top-1**) |

→ 多數圖表只是「把已算好的 Counter / history 輸出成陣列」,builder 改動很小。

---

## 2. 共用前端元件(Phase 1 先做)

新增三個無相依小函式(放 timeattack.js,純字串產生器):

- `renderBarChart(rows, opts)`:`rows = [{label, value, sub?, tone?}]` → 水平長條
  (CSS flex,寬度 ∝ value/max)。用於所有分布圖。
- `renderSparkline(points, opts)`:`points = [{x, y, mark?}]` → inline SVG 折線
  (含 PB 標點)。用於車手進步曲線。
- `renderDonut(parts)`:`parts = [{label, value, tone}]` → inline SVG 甜甜圈或堆疊條。
  用於 badge(TR/CR/PR)組成。

色票:主色 `--ta-accent`(車燈黃)、`--ta-accent-warm`(標誌綠)、`--ta-text-faint`,
最多 3~4 個 tone,避免彩虹化。CSS 加一段 `.ta-chart-*`。

## 3. 首頁(overview)— 熱門度頻率圖

**模組**:`renderModule("熱門度","Popularity","熱門賽道與車型","Busiest Tracks & Cars", …)`
接在 overview 現有模組(傳送門、近期紀錄)之後。

- 內容:**賽道熱門度水平長條**(依跑次降序,top 15 + 其餘合併「其他」),
  附一個 賽道 / 車型 切換(車型用 vehicle_run_counter)。
- 點長條 → 連到該賽道/車型詳情頁(`track.html?id=` / `vehicle.html?id=`)。
- builder:`build_overview_extras` 把現成的兩個 Counter 輸出成
  `popularity_chart: { tracks:[{code,name,count}], vehicles:[{model_code,name,count}] }`。
- 前端:`renderOverview` 加一個 `renderBarChart` 模組;切換用既有 data-view-toggle 機制或小按鈕。

## 4. 車手個人頁(player detail)— 尾端模組

接在 `renderPlayerDetail` 既有內容之後。資料幾乎都在 player card 現成欄位。

| 模組 | 圖 | 資料來源 | builder 改動 |
| --- | --- | --- | --- |
| **進步曲線(可切路線)** ★ | SVG sparkline(時間越短越好,標 PB)+ 路線下拉 | `history`(每條路線的時間序列,已存在) | 已有,**不預先攤平**,前端按選取即時繪 |
| **車種使用分布** | 水平長條 | `usage_rows`(vehicle Counter) | 已輸出 |
| **環境偏好** | 水平長條 | env Counter | 目前算了沒輸出 → 加 `env_rows` |
| **戰績組成 TR/CR/PR** | donut | `badge_counts` | 已有 |
| **活躍時間軸** | 月別長條(每月跑次) | 由 history/record rows 的 `record_date` 彙整 | 加 `monthly_activity:[{ym,count}]` |

### 4.1 進步曲線必須是互動式,不可預先攤平(重要修訂)

問題:車手會在很多條路線留紀錄,若 builder 把「每條路線的進步曲線」全部
預先 render 成靜態 SVG,markup 會爆量、DOM 也臃腫。

解法 — **資料已在 JSON,渲染按需求一次只畫一張**:

- `history` 早就帶了該車手「每條 ≥2 跑次路線」的逐次跑點(date/lap_time_ms/vehicle/is_pb),
  **不需要新增也不需要重複存**;爆量的只是「全部畫出來」,不是資料本身。
- 模組只放**一個路線下拉 + 一張 sparkline 容器**。預設選跑次最多的路線;
  使用者切下拉 → 前端從 `history` 取那條的 runs **即時重繪**,DOM 永遠只有一張圖。
- 下拉沿用詳情頁既有的「頁內切換列(filter + 下拉 + ◀▶)」互動語彙,風格一致。
- 因此 **builder 對進步曲線零改動**,純前端 `renderSparkline` + 一個 change 事件。
- players.json 體積估算:所有 history 跑點總和 ≤ 709 筆 × 約 5 欄,全站約數十 KB,可接受;
  若日後真嫌大,再用「只保留 ≥3 跑次路線」收斂,但目前不需要。

> 通則:**凡是「每條路線各一張」的圖,一律走下拉即時繪、不預先攤平**(見 §5 賽道頁同理可選)。

重點是**進步曲線**——車手最想看「自己在秋名下坡有沒有越來越快」。
sparkline 以 PB 連線 + 每次跑點呈現,滑過顯示日期/車輛/圈速。

## 5. 賽道個別頁(track detail)— 尾端模組

接在 `renderTrackDetail` 既有路線榜之後。該頁已握有此賽道每條路線的全部跑次。

| 模組 | 圖 | 資料來源 | builder 改動 |
| --- | --- | --- | --- |
| **車種使用分布** | 水平長條 | 此賽道全部 route_rows 的 vehicle_model_name Counter | 加 `vehicle_distribution` |
| **圈速分布(主路線)** | dot-strip 或直方圖 | 跑次最多路線的 route_rows lap_time_ms | 加 `laptime_spread:{route,times[],fastest,median}` |
| **競爭密度** | 數字卡 | 第一名與末名差距 %、有效跑次、不同車手數 | 由上者衍生 |
| **月活躍** | 月別長條 | route_rows 的 record_date | 加 `monthly_activity` |

「圈速分布」讓人一眼看出這條路線是「神仙打架(擠成一團)」還是「斷層」。

## 6. 車輛個體頁(vehicle detail)— 尾端模組

接在 `renderVehicleDetail` 既有內容之後。vehicle card 已有多個 Counter。

| 模組 | 圖 | 資料來源 | builder 改動 |
| --- | --- | --- | --- |
| **環境適性** | 水平長條 | env Counter(此車跑在 山道/高速/賽道…) | 加 `env_rows` |
| **車手分布** | 水平長條 | driver Counter | 已有 `tag_rows`(目前就是 driver) |
| **變體使用** | 水平長條 | variant Counter | 加 `variant_rows` |
| **路線涵蓋 / 競爭力** | 數字卡 + donut | 有跑路線數、其中 TR/CR 佔比(badge_counts) | 已有 |
| **同級對比(選配)** | 水平長條 | 此車 vs 同 vehicle_class 平均跑次/TR 率 | 加 class 級彙整 |

重點是**環境適性 + 路線涵蓋**:看出「這台是山道專武還是全能」、
「在多少條路線拿下 TR/CR」。同級對比較進階,列選配。

## 7. builder 改動清單(彙總)

- `build_overview_extras`:輸出 `popularity_chart`。
- `build_player_pages`:每張 card 加 `env_rows`、`monthly_activity`;`history` 已足夠進步曲線。
- `build_track_pages`:每個 board 加 `vehicle_distribution`、`laptime_spread`、`monthly_activity`。
- `build_vehicle_pages`:每張 card 加 `env_rows`、`variant_rows`(driver 已在 tag_rows);
  選配 class 級彙整。
- 一個共用小工具 `month_counter(records)` 產 `[{ym, count}]`,三頁共用。
- 其餘 JSON 不動;改完重跑 builder,既有頁輸出應 byte 不變(除了三張 detail JSON 與 summary)。

## 8. 前端改動清單(彙總)

- timeattack.js:加 `renderBarChart` / `renderSparkline` / `renderDonut` 三個工具;
  在 `renderTrackDetail` / `renderPlayerDetail` / `renderVehicleDetail` / `renderOverview`
  各自尾端 push 對應 `renderModule`。
- timeattack.css:加 `.ta-chart-bar` / `.ta-chart-spark` / `.ta-chart-donut` / `.ta-chart-grid` 樣式。
- 互動:長條可點(連到對應實體頁)、sparkline 點 hover tooltip;全部漸進增強,無 JS 也有靜態條。

## 9. 分期

| Phase | 內容 | 價值 |
| --- | --- | --- |
| **1** | 共用 3 工具 + CSS + **首頁熱門度長條**(資料現成) | 立即可上線,先驗證圖表風格 |
| **2** | 車手頁:進步曲線 + 車種/環境/badge | 最有「個人成長」感的一頁 |
| **3** | 賽道頁:車種分布 + 圈速分布 + 月活躍 | 賽道「戰況」一目了然 |
| **4** | 車輛頁:環境適性 + 車手 + 變體 + 涵蓋率(同級對比選配) | 車輛取向定位 |

每期完成定義:該頁分析模組顯示正確、小樣本有 fallback、builder 重跑其餘 JSON 不變、
新檔(若有)記得 git add。

## 10. 風險與守則

- **小樣本**:很多車手/車型只有 1~2 筆;一律設門檻 + 「資料太少」字樣,不畫誤導圖。
- **日期稀疏**:`record_date` 有缺漏時月活躍要容錯(跳過空值、軸自動收斂)。
- **CJK in SVG**:SVG `<text>` 要指定字型且檔案 UTF-8;終端驗證一律 Read 工具。
- **效能**:純前端聚合也行,但堅持 builder 端預算,避免重複邏輯散落 JS。
- **設計克制**:每頁最多 3~4 個模組,圖表配色 ≤ 4 色,維持暗色系一致性。
